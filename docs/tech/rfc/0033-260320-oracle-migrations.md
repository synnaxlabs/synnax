# 33 - Oracle Migration System

**Feature Name**: Oracle Migration System <br /> **Status**: In Progress <br /> **Start
Date**: 2026-02-22 <br /> **Authors**: Emiliano Bonilla <br />

# 0 - Summary

A standardized migration system integrated with Oracle's schema-first code generation.
The system has two layers: a general-purpose migration framework in `x/go/gorp` that any
gorp user can leverage, and an Oracle code generation layer that produces frozen type
snapshots, frozen codecs, and migration functions that plug into gorp's infrastructure.

When Oracle schemas change, migrations are generated that transform stored data from the
old schema to the new one. Each schema version gets its own `migrations/vN/` sub-package
containing a frozen copy of the type at that version. The migration chain maps entirely
within the `migrations/` subtree (`v1.TypeV1 -> v2.TypeV2 -> ...`), never importing the
parent service package. The developer's only responsibility is deciding what new fields
should default to for existing data.

# 1 - Vocabulary

- **Oracle** - Schema-first code generation tool that compiles `.oracle` files into Go,
  TypeScript, Python, C++, and Protobuf types.
- **gorp** - Type-safe ORM wrapping Pebble KV store with `Reader[K, E]` and
  `Writer[K, E]` generics. Stores entries under a canonical KV prefix derived from the
  Go type name (e.g., `__gorp__//Schematic`).
- **Table** - `gorp.Table[K, E]` manages codec-aware reads/writes and runs versioned
  migrations at startup via `OpenTable`. Replaces the earlier `EntryManager` concept.
- **Migration** - A versioned transform that converts entries from one schema version to
  the next during server startup.
- **Auto-migrate** - Oracle-generated helper function that copies all 1:1 fields from
  the old type to the new type. Never edited by the developer. The developer calls this
  helper explicitly from within their transform function.
- **Transform** - Oracle-generated template called for each entry during migration. The
  developer calls the auto-migrate helper and sets default values for new fields.
- **KV prefix** - The key prefix under which gorp stores all entries of a type (e.g.,
  `__gorp__//Schematic`). Derived from the Go type name via `types.Name[E]()`.
- **Codec** - Serialization format used to encode/decode entries in the KV store.
  Currently transitioning from msgpack to ORC binary encoding via Oracle-generated
  `EncodeOrc`/`DecodeOrc` methods on entry types.
- **ORC** - Oracle Runtime Codec. A binary encoding format with a 3-byte magic header
  (`0x4F, 0x52, 0x43`). Types opt in by implementing
  `orc.SelfEncoder`/`orc.SelfDecoder`. The ORC codec wraps a fallback codec (msgpack)
  and dispatches based on whether the type implements these interfaces.
- **Frozen type** - A snapshot of a type's struct definition at a previous schema
  version, stored in `migrations/vN/`. Each frozen version has its own codec with type
  assertions matching the frozen type.
- **Snapshot** - A copy of `.oracle` source files at the time of a migration generation,
  stored in `schemas/.snapshots/<version>/` for CI diffing.

# 2 - Motivation

Several Oracle-managed types (schematics, workspace layouts, device configurations) are
stored as loosely typed blobs in the KV store. As the platform matures, these types need
stronger typing, and the storage encoding needs to transition from msgpack to direct
binary encoding. Both changes require a systematic way to evolve stored data without
data loss or downtime.

The goals of this migration system are:

- Standardize migration tooling across all Oracle-managed types
- Strongly type server-side data structures (schematics, workspace layouts, etc.)
- Support automatic migration generation when Oracle schemas change
- Allow custom migration logic defined by the developer
- Enable the transition from msgpack to binary encoding for storage

The first migration to run through the system is the codec switch from msgpack to binary
for gorp-stored entries. This is a high-value change that exercises the core migration
pipeline (iterate all entries, decode with old codec, re-encode with new codec) without
requiring schema changes to any individual type. It validates the infrastructure before
more complex per-type schema migrations are needed.

# 3 - Principles

## 3.0 - The Type Knows Its Codec

A codec is not an external dependency. It is a derived property of the type. Given the
struct definition of `Schematic`, there is exactly one correct binary encoding. The type
carries its own codec via Oracle-generated `EncodeOrc`/`DecodeOrc` methods. No
per-service codec wiring. No `Codec` field in `ServiceConfig`.

```go
// Generated by oracle alongside types (in codec.gen.go)
func (s Schematic) EncodeOrc(w *orc.Writer) error { /* ... */ }
func (s *Schematic) DecodeOrc(r *orc.Reader) error { /* ... */ }
```

The dispatch happens at the DB level. The distribution layer sets a single ORC codec
with msgpack fallback via `gorp.WithCodec(orc.NewCodec(msgpack.Codec))`. On encode, the
ORC codec checks whether the value implements `orc.SelfEncoder`. If so, it writes binary
with a magic header. If not, it falls back to msgpack. On decode, it checks for the
magic header to decide which path to take. This means types opt into binary encoding by
implementing the interface, with no per-service configuration needed.

**Tradeoff acknowledged**: The codec is less visible than explicit DI. You cannot see
which codec a service uses by reading `layer.go`. The tradeoff is accepted because: (a)
the codec IS intrinsic to the type, not a configurable choice, (b) the ORC fallback
handles mixed states gracefully during transition, and (c) the alternative was 12 lines
of identical codec wiring in `layer.go` that added no information.

## 3.1 - Nested Types Are Not Independent

Nested types (like `types.Param`, `graph.Node`, `ir.Edge`) have no independent storage.
They exist only as fields within gorp entries. They do not have migration chains. They
do implement `EncodeOrc`/`DecodeOrc`, but only because the parent's codec calls them
during encoding. When a nested type changes, the parent entry's migration handles it.

## 3.2 - Each Migration Step Is Self-Contained

A migration is self-contained. It receives a `gorp.Tx` that carries the DB's codec (ORC
with msgpack fallback). The ORC codec handles format detection automatically via the
magic header: old msgpack data is decoded via the fallback, new writes go out as binary.
Frozen binary codecs are available for post-transition steps that need to decode a
specific historical format.

## 3.3 - Migrations Are Per-Entry Transforms (With Escape Hatch)

The primary migration model is a function on a single entry: decode one, transform it,
write it back. This covers the vast majority of cases (field additions, removals, type
changes, codec transitions). For complex cases that need cross-type context (e.g.,
ontology restructuring), the developer uses `NewMigration` which provides a full
`gorp.Tx` for arbitrary read/write operations.

## 3.4 - The Developer Writes Only What Requires Human Judgment

Oracle generates everything mechanical: frozen type snapshots, frozen codecs,
auto-migrate helper functions (field copying), migration registration. The developer
writes only the transform function, which calls the generated auto-migrate helper and
sets default values for new fields.

## 3.5 - Safety Comes from Tests, Not Manual Review

The auto-migrate helper is generated code. Oracle generates golden tests alongside each
migration: seed known old data, run the migration, assert the output matches expected
values. The developer reviews test expectations (compact, understandable) rather than
generated field-copy code (verbose, mechanical). If the auto-migrate has a bug, the
golden test catches it.

## 3.6 - Frozen Snapshots Are Complete and Self-Contained

Each migration version duplicates the complete type tree as frozen copies within the
migrations package. The migrations package never imports the parent service package,
eliminating circular dependencies. Frozen types for `ArcV1` include all nested types
(`ParamV1`, `FunctionV1`, etc.) at their v1 definitions. Each frozen version has its own
codec that type-asserts on the correct frozen type.

# 4 - Design

## 4.0 - Two-Layer Architecture

```
Oracle (build time)              gorp.OpenTable (runtime)
  |                                |
  +-- oracle sync                  +-- reads __gorp_migration__//<Type>
  |   +-- generates types.gen.go   +-- topologically sorts pending migrations
  |   +-- generates codec.gen.go   +-- runs pending migrations in order:
  |      (EncodeOrc/DecodeOrc)     |   +-- EntryMigration: decode -> transform -> encode
  |                                |   +-- RawMigration: arbitrary gorp.Tx logic
  +-- oracle migrate               +-- persists applied migration names
  |   +-- generates vN/ snapshots  +-- (built-in) normalize_keys
  |   +-- generates auto-migrate
  |   +-- generates transform template
  |   +-- generates vN/codec.gen.go (frozen EncodeOrc/DecodeOrc)
  |   +-- generates migrate.gen.go
  |
  +-- oracle migrate check
      +-- CI enforcement
```

**Layer 1 (gorp)**: A migration framework in `x/go/gorp` that any gorp user can use.
Provides the `Migration` interface, `OpenTable` with name-based version tracking,
`NewEntryMigration`, `NewMigration`, dependency-based ordering via `WithDependencies`,
and transactional execution. Knows nothing about Oracle. Cesium, pebblekv, and
hand-written migrations all use this layer directly.

**Layer 2 (Oracle)**: Built on top of the gorp layer. Oracle owns schemas, generates
frozen type snapshots, generates frozen codecs, and generates migration functions that
implement `gorp.Migration`. The `oracle migrate` CLI produces files that plug into
gorp's infrastructure.

## 4.1 - Codec Architecture

### 4.1.0 - encoding.Codec Is the Interface

The platform's universal serialization interface is `encoding.Codec` in `x/go/encoding`.
No separate gorp-specific codec abstraction.

```go
type Codec interface {
    Encode(ctx context.Context, value any) ([]byte, error)
    Decode(ctx context.Context, data []byte, value any) error
}
```

### 4.1.1 - ORC Codec with Fallback Dispatch

Codec resolution is a DB-level concern, not per-table or per-type. The distribution
layer creates a single `orc.NewCodec(msgpack.Codec)` and passes it to `gorp.WithCodec()`
when wrapping the KV store.

```go
// core/pkg/distribution/layer.go
l.DB = gorp.Wrap(aspenDB, gorp.WithCodec(orc.NewCodec(msgpack.Codec)))
```

The ORC codec dispatches based on interface implementation:

- **Encode**: If value implements `orc.SelfEncoder`, call `EncodeOrc(w)` and prepend the
  3-byte magic header (`0x4F, 0x52, 0x43`). Otherwise, fall back to msgpack.
- **Decode**: If data starts with the magic header, strip it and call `DecodeOrc(r)`.
  Otherwise, fall back to msgpack.

This means old msgpack data and new binary data coexist in the same KV store. No
flag-day cutover needed. Types that implement `SelfEncoder`/`SelfDecoder` (via
Oracle-generated `codec.gen.go`) automatically get binary encoding on write and can
still read their old msgpack entries.

### 4.1.2 - Migration Codecs Are Inherited from the DB

Migrations receive a `gorp.Tx` which embeds the DB's `encoding.Codec`. The Tx uses the
same ORC-with-msgpack-fallback codec as the rest of the system. This avoids the problem
of needing separate migration codecs: the ORC codec handles mixed-format data
transparently via the magic header check.

### 4.1.3 - Type Assertions in Generated Codecs

Generated codecs use Go type assertions: `value.(Schematic)`. A codec generated for
`schematic.Schematic` will not work for `v1.Schematic` from a different package, even if
the binary layout is identical.

**Consequence**: Each frozen type version must have its own generated codec. Oracle
generates `v1.SchematicCodec` that asserts `value.(v1.Schematic)`, separate from the
current `SchematicCodec` that asserts `value.(Schematic)`.

The codec transition (msgpack to binary) does not need frozen types because it uses the
current type for both input and output. MsgPack decodes by struct tags (name-based).
Binary encodes by field position. Both work with the current type.

### 4.1.4 - Binary Encoding Format

Oracle-generated codecs use direct binary encoding via `orc.Writer`/`orc.Reader`. The
encoding is **positional**: field order in the `.oracle` schema determines wire layout.
The ORC writer handles byte ordering, length-prefixed strings, and nested type encoding.

```go
// Oracle-generated codec (in codec.gen.go, same package as the type)
func (s Schematic) EncodeOrc(w *orc.Writer) error {
    w.Write(s.Key[:])                  // uuid: 16 bytes
    w.String(s.Name)                   // string: len-prefixed
    w.JSON(s.Data)                     // json: len-prefixed
    w.Bool(s.Snapshot)                 // bool: 1 byte
    return nil
}

func (s *Schematic) DecodeOrc(r *orc.Reader) error {
    var err error
    if _, err = r.Read(s.Key[:]); err != nil { return err }
    if s.Name, err = r.String(); err != nil { return err }
    if err = r.JSON(&s.Data); err != nil { return err }
    if s.Snapshot, err = r.Bool(); err != nil { return err }
    return nil
}
```

## 4.2 - Migration Model

### 4.2.0 - Two Migration Types at the gorp Layer

**EntryMigration[IK, OK, I, O]**: Iterates all entries, decodes each as type `I`, calls
a transform function to produce type `O`, writes back. Both decoding and encoding use
the DB's codec from the transaction.

```go
func NewEntryMigration[IK Key, OK Key, I Entry[IK], O Entry[OK]](
    name string,
    transform func(ctx context.Context, old I) (O, error),
) Migration
```

The transform receives `context.Context` for cancellation and tracing during long
migrations. Logging is handled by the framework via `alamos.Instrumentation`.

At the Oracle layer, the auto-migrate is a generated helper function that the developer
calls explicitly:

```go
// migrate_auto.gen.go -- generated helper, copies all matching fields
func AutoMigrateSchematic(ctx gorp.MigrationContext, old v1.Schematic) (Schematic, error) {
    return Schematic{Key: old.Key, Name: old.Name, Data: old.Data, Snapshot: old.Snapshot}, nil
}

// migrate.go -- developer writes this
func MigrateSchematic(ctx context.Context, old v1.Schematic) (Schematic, error) {
    new := AutoMigrateSchematic(ctx, old)
    new.Description = ""
    return new, nil
}
```

**RawMigration**: Receives a fully wrapped `gorp.Tx` with the DB's codec. Full control
over reads and writes. Used for cross-type migrations like ranger's group restructuring.

```go
func NewMigration(
    name string,
    fn func(ctx context.Context, tx Tx, ins alamos.Instrumentation) error,
) Migration
```

A separate `CodecTransition` type is not needed. The ORC codec's magic-header dispatch
handles mixed-format data transparently: old msgpack entries are decoded via the
fallback, and writes go out as binary for types implementing `SelfEncoder`. A no-op
`EntryMigration` (identity transform) is sufficient to re-encode all entries.

### 4.2.1 - Two-Function Pattern

Every migration has two functions in two files: an auto-migrate helper (generated, never
touched) and a transform (template, developer edits).

**Auto-migrate** (`migrate_auto.gen.go`): Oracle-generated, regeneratable, **never
edited by the developer**. Handles all mechanical work: copying 1:1 fields from old to
new type, calling nested auto-migrations, walking arrays of nested types. This file can
be regenerated at any time without losing work.

**Transform** (`migrate.go`): Oracle-generated **once** as a template, then owned by the
developer. Calls the auto-migrate helper and adds custom logic: setting default values
for new fields, computing derived values, transforming data. Oracle pre-populates this
with TODOs for fields it cannot infer.

The runtime calls them in sequence for each entry:

```
old entry -> AutoMigrate (generated helper) -> developer transform -> new entry
```

| File                  | Generated by  | Regeneratable | Developer edits |
| --------------------- | ------------- | ------------- | --------------- |
| `migrate_auto.gen.go` | Oracle        | Yes (always)  | **Never**       |
| `migrate.go`          | Oracle (once) | No            | **Yes**         |

This separation means Oracle can improve its auto-migration logic and regenerate
`auto.gen.go` files without clobbering developer work in `migrate.go`.

### 4.2.2 - Three Generation Modes

Oracle has three generation modes depending on the nature of the change:

- **Full mode**: Mechanical migrations (codec transitions). Both auto-migrate and
  transform are generated. Developer touches neither file.
- **Skeleton mode**: Direct schema changes (field add/remove/rename/type change).
  Auto-migrate copies 1:1 fields. Transform template has TODOs for changed fields.
  Developer edits the transform.
- **Propagation mode**: A nested dependency changed. Parent auto-migrate walks the tree
  and calls nested auto-migrate + transform. Parent transform template is empty.
  Developer only edits the leaf transform.

### 4.2.3 - Migration Execution

On server startup, `OpenTable` runs pending migrations in a single KV transaction:

1. Read applied migrations from `__gorp_migration__//<Type>` (JSON array of migration
   name strings)
2. Prepend built-in `normalize_keys` migration, wrap user migrations to depend on it
3. Topologically sort pending migrations (those not in the applied set) using declared
   dependencies
4. For each pending migration: execute `Migration.Run(ctx, tx, instrumentation)`
5. After each migration, add its name to the applied set and persist back to KV
6. Commit transaction atomically
7. If any step fails, entire transaction rolls back. Retry on next startup.

Key normalization (`normalize_keys`) is a built-in migration that runs before all user
migrations. It handles prefix format changes (msgpack-encoded prefix to raw string
prefix) and key re-encoding. All user migrations implicitly depend on it via the
`afterBuiltIn` wrapper. It is a cheap no-op on subsequent startups.

Migration ordering uses dependency-based topological sort rather than sequential version
numbers. Each migration declares its dependencies by name via the `DependencyDeclarer`
interface. Oracle generates these dependencies automatically (each migration depends on
the previous one in the chain). The graph library detects cycles and missing
dependencies.

### 4.2.4 - Sequential Chaining

Migrations run in sequential order (v1->v2->v3). Each migration handles a single version
step. When a customer skips versions (e.g., v0.48 -> v0.51), all intermediate migrations
execute in order.

No rollback support. Migrations are tested before shipping to customers. Development
data is disposable, so broken migrations during development are fixed by wiping local
data and re-running. If a bad migration ships to customers, a corrective migration is
released in a subsequent version.

When multiple migrations are pending, the current design runs each as a full pass over
all entries:

- **v1->v2**: Iterate all entries, decode as V1, transform to V2, encode, write.
- **v2->v3**: Iterate all entries again, decode as V2, transform to V3, encode, write.

An optimization would be to chain transforms in memory (decode once, apply all
transforms, encode once, write once). However, this complicates the runner because (1)
intermediate types exist only at compile time, and (2) raw migrations need KV state from
the previous step. Multiple full passes is the design for now. Simple, correct, and fast
enough for metadata volumes (thousands of entries, not millions).

### 4.2.5 - Multi-Node Coordination

Each node in a Synnax cluster runs migrations on its own local KV store during startup.
No leader election or distributed locking. This works because Aspen gossip replicates
metadata and each node holds a local copy that it can migrate independently. Migrations
must be deterministic and idempotent: given the same input data, every node must produce
the same output.

## 4.3 - Code Generation

### 4.3.0 - Migrations Package Structure

Migration artifacts are co-located with their service package. The top-level
`migrate.gen.go` sits alongside `service.go`. When schema changes occur, frozen types go
into `migrations/vN/` sub-packages that never import the parent service package.

```
core/pkg/service/schematic/
    types.gen.go           # current types (oracle sync)
    codec.gen.go           # current codec (oracle sync, EncodeOrc/DecodeOrc)
    migrate.gen.go         # SchematicMigrations() -> []gorp.Migration (generated)
    migrate.go             # developer transform for latest migration
    migrate_auto.gen.go    # auto-migrate helper for latest migration
    service.go             # service logic
    migrations/
        v1/
            types.gen.go   # frozen type snapshot at v1
            codec.gen.go   # frozen codec for v1
```

Before any schema changes, `migrate.gen.go` returns an empty slice. Oracle populates it
when migrations are generated.

**Key structural rules:**

- Each `vN/` is a separate Go package. Migration functions map `vN.Type -> v(N+1).Type`
  (or `vN.Type -> parentpkg.Type` for the final step). They never import the parent
  service package.
- The last migration step targets the current type in the parent package directly (e.g.,
  `v1.Schematic -> schematic.Schematic`), not a frozen copy.
- Pre-codec-transition versions (e.g., `v1/`) decode using the DB's ORC codec with
  msgpack fallback. The ORC magic header check handles format detection automatically.
- Post-codec-transition versions include a frozen `codec.gen.go` with
  `EncodeOrc`/`DecodeOrc` methods. Each version's codec is a snapshot of the binary
  layout at that version, write-once, never regenerated.
- The parent package's current type (in `types.gen.go`) is structurally identical to the
  latest schema. Oracle generates both from the same `.oracle` source.

### 4.3.1 - Migration Registration

`migrate.gen.go` is Oracle-generated and regenerated on every `oracle migrate` run. It
exports a `<Type>Migrations()` function that returns an ordered `[]gorp.Migration` list
with dependency declarations. Oracle maintains it as the single source of truth for
migration ordering.

```go
// service.go
table, err := gorp.OpenTable[uuid.UUID, Schematic](ctx, gorp.TableConfig[Schematic]{
    DB:              cfg.DB,
    Migrations:      SchematicMigrations(),
    Instrumentation: cfg.Instrumentation,
})
```

Services can append custom migrations alongside generated ones:

```go
Migrations: append(RangeMigrations(), &groupMigration{cfg: cfg}),
```

Applied migrations are tracked at `__gorp_migration__//<Type>` as a JSON array of
migration name strings. The DB's codec is ORC with msgpack fallback, handling both
pre-transition and post-transition data transparently.

## 4.4 - Binary Transition

### 4.4.0 - Codec Transition Scope

The goal is a full codec replacement for all Oracle-managed types stored via gorp:
switch from msgpack to direct binary encoding. Since Oracle owns the schema and
generates both msgpack-compatible Go structs and binary codecs, the codec transition is
Oracle-generated rather than manually written per-type.

### 4.4.1 - Field Ordering Stability

Binary encoding is positional. Unlike protobuf (where field numbers provide tagged
access), reordering fields in a binary-encoded schema changes the wire format and makes
existing data unreadable.

This is safe because each migration version freezes a `codec.gen.go` that exactly
matches its type snapshot. Version N's data is always decoded with version N's frozen
codec, never with version N+1's codec. Oracle can freely add or remove fields between
versions because it generates a new frozen codec for each version.

**Safety rules:**

- Never reorder fields in an existing `.oracle` schema. Append new fields at the end.
- Frozen codecs are write-once. `vN/codec.gen.go` is generated once and never
  regenerated.
- `oracle migrate check` should detect field reordering as an error.
- Round-trip golden tests should encode a known value and assert the byte output matches
  a frozen expectation.

### 4.4.2 - JSON Fields in Binary Encoding

JSON fields (`Schematic.data`, `Workspace.layout`) are encoded using `encoding/json` and
stored as length-prefixed byte blobs within the binary format. This preserves full JSON
fidelity. When these fields are eventually promoted to full Oracle types, they get
proper binary encoding with correct numeric types.

### 4.4.3 - Startup Sequence After Binary Switch

```
1. gorp.OpenTable[K, Schematic](ctx, TableConfig{
       DB:              db,
       Migrations:      SchematicMigrations(),
       Instrumentation: cfg.Instrumentation,
   })
   // DB codec is orc.NewCodec(msgpack.Codec), set once at distribution layer

2. OpenTable runs normalize_keys (built-in, no-op if already done)

3. OpenTable runs pending migrations:
   v1_schema_migration: iterates all entries under prefix
     ORC codec detects msgpack data (no magic header), falls back to msgpack decode
     transforms v1.Schematic -> Schematic (current type)
     ORC codec encodes via Schematic.EncodeOrc(), writes with magic header
     writes back to KV

4. Service accepts requests
   all reads: ORC codec detects magic header, uses DecodeOrc
   all writes: ORC codec uses EncodeOrc, writes with magic header
   msgpack fallback still available but not exercised for migrated types
```

After migration completes, all entries for that type have ORC magic headers. The
fallback path is still present but idle. This is safe because the magic header check is
cheap (3-byte comparison) and the fallback handles any entries that were missed.

## 4.5 - Schema Change Detection

### 4.5.0 - Manual Generation with CI Enforcement

The developer manually runs `oracle migrate` to create migration files when they change
a schema. Oracle stores a snapshot of the schema state at the time of the last migration
generation. CI runs `oracle migrate check` which validates that the current `.oracle`
files match the latest snapshot. If the schema has changed but no migration has been
generated, CI fails.

All schema changes require a migration entry. Any modification to an Oracle schema
(adding a field, removing a field, renaming, changing a field's type, changing
required/optional status) requires a corresponding migration. Explicit is better than
implicit. Even additive changes that are technically safe with msgpack deserialization
get a migration entry. This ensures a complete audit trail and prevents surprises when
the storage codec changes.

### 4.5.1 - Schema Snapshots

When `oracle migrate` runs, Oracle copies the current `.oracle` source files into a
versioned snapshot directory (e.g., `schemas/.snapshots/v3/schematic.oracle`). CI diffs
current `.oracle` files against the latest snapshot. If they differ and no new migration
has been generated, CI fails. Legacy types are regenerated from the snapshot by
re-running Oracle's parser on the old `.oracle` source.

This approach is human-readable, git-diffable, and uses Oracle's own source format as
the authoritative record. Storage is cheap (<200KB per snapshot). Developers can inspect
snapshots directly to understand what changed between migration versions.

For the first migration (codec transition): no previous snapshot needed. The old schema
IS the current schema (same fields, different encoding). The snapshot is created after
the first migration generation, establishing the baseline.

### 4.5.2 - CLI Interface

**`oracle migrate`**: Primary command for schema changes. Generates migration artifacts,
then runs `oracle sync` internally to update types/codecs. One command.

```bash
# 1. Edit .oracle schema
# 2. Run oracle migrate (generates migrations + runs sync)
oracle migrate
# 3. Edit the transform file (set defaults for new fields)
# 4. Run tests, commit
```

Behavior:

1. Diffs current `.oracle` files against the latest snapshot in
   `schemas/.snapshots/<version>/`.
2. For each changed type, determines the generation mode:
   - **Full mode**: Codec transitions. Both files fully generated.
   - **Skeleton mode**: Direct schema changes. Auto-migrate copies unchanged fields.
     Transform template has TODOs.
   - **Propagation mode**: Nested dependency changed. Parent auto-migrate walks the
     tree.
3. Generates files into the appropriate `migrations/` directories.
4. Regenerates `migrate.gen.go` for each affected service.
5. Creates a new snapshot in `schemas/.snapshots/<version+1>/`.

**`oracle sync`**: Standalone for non-migration tasks (initial setup, Oracle upgrades,
regenerating after toolchain changes). Does not generate migrations.

**`oracle migrate check`**: CI enforcement. Diffs current schemas against latest
snapshot. Fails if schemas changed without corresponding migration.

```yaml
- name: Check Oracle migrations
  run: oracle migrate check
```

## 4.6 - Nested and Shared Type Migrations

### 4.6.0 - Dependency Tracking

Oracle's transitive dependency tracking detects when a type references another Oracle
type. When type B changes and type A contains a field of type B, Oracle flags that a
migration is needed for type A as well. The CI check fails if type B changes but type A
doesn't have a corresponding migration.

### 4.6.1 - Nested Type Propagation

When a nested type changes (e.g., `graph.Node` gains a `label` field), Oracle's
dependency graph detects that a gorp entry (e.g., `Arc`) contains `graph.Node`. Oracle:

1. Bumps the gorp entry's version.
2. Generates the auto-migrate for the parent that walks the tree and calls Node's
   auto-migrate and transform on each node.
3. Generates an empty transform template for the parent (in case the developer needs
   custom parent-level logic too).

**Parent auto-migrate** (`core/pkg/service/arc/migrations/v5/auto.gen.go`):

```go
// GENERATED BY ORACLE -- DO NOT EDIT
package v5

import (
    v6 "core/pkg/service/arc/migrations/v6"
    graphv1 "schemas/arc/graph/migrations/v1"
    graphv2 "schemas/arc/graph/migrations/v2"
)

func AutoMigrateV5ToV6(ctx context.Context, old ArcV5) (v6.ArcV6, error) {
    nodes := make([]graphv2.NodeV2, len(old.Graph.Nodes))
    for i, n := range old.Graph.Nodes {
        migrated, err := graphv1.AutoMigrateV1ToV2(ctx, n)
        if err != nil { return v6.ArcV6{}, err }
        if err := graphv1.PostMigrateV1ToV2(ctx, &migrated, n); err != nil {
            return v6.ArcV6{}, err
        }
        nodes[i] = migrated
    }
    return v6.ArcV6{
        Key:    old.Key,
        Name:   old.Name,
        Mode:   old.Mode,
        Graph:  v6.GraphV6{
            Viewport:  old.Graph.Viewport,
            Functions: old.Graph.Functions,
            Edges:     old.Graph.Edges,
            Nodes:     nodes,
        },
        Text:   old.Text,
        Module: old.Module,
        Status: old.Status,
    }, nil
}
```

For a pure nested propagation, the developer doesn't touch the parent transform at all.

### 4.6.2 - Shared Nested Types

When a nested type is referenced by multiple gorp entries (e.g., `ir.Edge` used by both
`Arc` and `Schematic`), Oracle generates parent auto-migrations for every entry that
contains the changed type. All generated auto-migrations call the same leaf auto-migrate
and transform functions. The leaf transform logic is written once.

### 4.6.3 - Execution Order

Nested type migrations don't execute independently. They are called within the parent
entry's auto-migrate. Only gorp entries (types with KV prefixes) have their own
migration runner. Nested types are migrated as part of their parent's
read-transform-write cycle.

Packages don't have circular dependencies. Services start in dependency order. If
Workspace depends on Schematic, Schematic's service (and its migrations) runs first. By
the time Workspace's migrations execute, everything Workspace depends on is already
migrated. No additional ordering mechanism needed.

## 4.7 - Strongly Typing JSON Fields

### 4.7.0 - Incremental Adoption

The end goal is to define the complete nested structure of complex fields (schematic
data, workspace layout, etc.) as Oracle types. Oracle generates Go/TS/Python types for
maximum type safety.

The adoption path is incremental: start by defining the top-level structure as an Oracle
type while nested fields remain `json`/`any`. Then use the migration system itself to
progressively promote nested fields to strongly typed Oracle types. Each promotion step
is a schema change + migration.

### 4.7.1 - Unknown JSON Keys

When promoting a `json` field to a struct, unknown keys are silently dropped. The
migration only copies keys that map to struct fields.

1. **The server is adopting the client's type definition.** The client already ignores
   unknown keys when deserializing. The server should behave the same way.
2. **Unknown keys are noise, not data.** They come from deprecated fields, experimental
   client features, or typos. Preserving them adds complexity with no value.
3. **No catch-all field.** A `map[string]any` overflow field defeats the purpose of
   strong-typing and creates a permanent escape hatch that never gets cleaned up.
4. **If specific data needs preserving**, the developer can explicitly copy it in the
   transform function before the migration drops it.

# 6 - Testing

## 6.0 - Test Helpers

The migration framework provides test utilities that reduce boilerplate. The primary
helper is `oracle.TestMigration` which handles setup, execution, and assertion in a
single call.

```go
var _ = Describe("SchematicV2ToV3", func() {
    It("Should set description to nil for existing entries", func() {
        oracle.TestMigration(
            // Old entries to seed into the KV store.
            []v2.SchematicV2{
                {Key: uuid.New(), Name: "My Schematic", Data: someJSON},
            },
            // Expected new entries after migration.
            []v3.SchematicV3{
                {Key: /* same */, Name: "My Schematic", Data: someJSON, Description: nil},
            },
            // The migration to run.
            oracle.RegisterTyped(
                "add_description",
                v2.AutoMigrateV2ToV3,
                v2.PostMigrateV2ToV3,
            ),
        )
    })
})
```

**What `TestMigration` does internally**:

1. Creates an in-memory KV store
2. Encodes and writes old entries under the canonical prefix
3. Runs the migration function
4. Reads back all entries, decodes as the new type
5. Asserts results match expected entries (using `gomega.Equal`)
6. Validates the migration was recorded in the applied set

**Testing nested migrations**:

```go
var _ = Describe("NodeV1ToV2 (nested)", func() {
    It("Should set label to empty string for existing nodes", func() {
        old := v1.NodeV1{Key: "node-1", Type: "opc", Position: spatial.XY{X: 10, Y: 20}}
        migrated, err := v1.AutoMigrateV1ToV2(ctx, old)
        Expect(err).ToNot(HaveOccurred())
        Expect(v1.PostMigrateV1ToV2(ctx, &migrated, old)).To(Succeed())
        Expect(migrated.Label).To(Equal(""))
    })
})
```

Nested type migrations are tested as pure functions (no KV store needed) since they
execute within the parent's migration runner.

# 7 - Scope and Boundaries

## 7.0 - Non-Oracle Types

Non-Oracle gorp-stored types (Cesium internals, Aspen KV metadata, ontology resources)
retain their existing migration mechanisms. This RFC covers only Oracle-managed types.

## 7.1 - Migration Tracking Storage

Each Oracle-managed type gets a dedicated KV key (e.g., `__gorp_migration__//Schematic`)
that stores a JSON array of applied migration names. All entries of that type are
assumed to be at the same version after startup migration completes. No version field is
added to individual entries.

## 7.2 - Relationship to RFC 0025

The Oracle migration system builds the server-side infrastructure for schema evolution.
RFC 0025 (moving client-side migrations to the server) will layer on top once server
types are strongly typed via Oracle. The sequence is:

1. **This RFC (0033)**: Oracle migration system + msgpack->binary codec transition
2. **Strongly type JSON fields**: Incrementally promote `json` fields to Oracle types
3. **RFC 0025**: Server owns all schema evolution; client sends/receives Oracle-typed
   data instead of managing its own migrations

| #   | Decision                                         | Rationale                                                                     |
| --- | ------------------------------------------------ | ----------------------------------------------------------------------------- |
| 1   | ORC SelfEncoder/SelfDecoder on entry types       | Codec is intrinsic to type. DB-level dispatch via magic header.               |
| 2   | Nested types don't carry codecs                  | No independent storage. Parent owns encoding.                                 |
| 3   | Per-entry transform primary, raw escape hatch    | Covers 95% of cases. Raw for complex cross-type ops.                          |
| 4   | Two-file split (auto.gen + developer transform)  | Safety from golden tests, not manual review.                                  |
| 5   | Two-layer architecture (gorp + Oracle)           | gorp is general-purpose. Oracle builds on it.                                 |
| 6   | Auto-migrate as helper function                  | Developer calls explicitly. Stack-based (value in, value out).                |
| 7   | Transform keeps context.Context                  | Operational need: cancellation, tracing, logging for long migrations.         |
| 8   | ORC fallback replaces CodecTransition            | Magic header dispatch handles mixed-format data without explicit re-encoding. |
| 9   | Key normalization as built-in migration          | Runs first via dependency declaration. Cheap no-op on subsequent startups.    |
| 10  | Retry on failure, nothing more                   | Server refuses to start. Fix and restart.                                     |
| 11  | oracle migrate runs sync internally              | One command. Correct ordering guaranteed.                                     |
| 12  | Implicit cross-type ordering via service startup | Service dependency order handles it.                                          |
| 13  | Single transaction, no batching                  | Metadata volumes are small. Simple and correct.                               |
| 14  | Migrations sub-package, self-contained           | Never imports parent. Each version has own codec.                             |
| 15  | Snapshot: full .oracle file copy per migration   | Human-readable, git-diffable, cheap storage.                                  |

# 10 - Open Questions

1. **Testing strategy**: Whether to keep or refine the `oracle.TestMigration` helper
   design. Deferred until the system is working end-to-end.
2. **Full re-encode**: Whether to ship an explicit re-encode migration (no-op
   `EntryMigration`) per type to eliminate all msgpack data from disk, or let the ORC
   fallback handle it indefinitely.
3. **Dependency injection in raw migrations**: Scenarios 5.5 and 5.6 show cases where
   migrations need access to other services. The current `gorp.Tx` escape hatch provides
   KV access but not service-level dependencies. May need a migration context with
   injectable services.
